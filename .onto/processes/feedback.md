# Domain Feedback Loop Process

> Transforms accumulated learnings into domain document improvement suggestions.
> Related: Learnings are generated by `.onto/processes/review.md`. Seeds are created by `.onto/processes/create-domain.md`. Promotion is handled by `.onto/processes/learn/promote-domain.md`.

Collects learnings tagged with `[domain/{domain}]`, compares them against the current domain document content, generates change suggestions, and applies user-approved changes. Works on both `drafts/` and `domains/` locations.

**Learning atomicity assumption**: This process assumes each learning is atomic (one independent claim per entry). See design document Section 4-8 for the empirical basis and contingency plan.

**Compare-based deduplication**: This process does not use state markers on learnings. Deduplication is determined by semantic comparison against the current document content at execution time (design document Section 4-4). This ensures correctness even after user direct edits.

### 0. Domain Resolution

Resolve the target domain location:

1. Check `~/.onto/drafts/{domain}/` — if exists, use as target.
2. If not in drafts, check `~/.onto/domains/{domain}/` — if exists, use as target.
3. If neither exists, report error and halt.

Also collect learning sources:
- Global learnings: `~/.onto/learnings/`
- Project learnings: `{project}/.onto/learnings/`

### 1. Learning Collection

From all learning sources (global + project), collect entries tagged with `[domain/{domain}]`.

Exclude entries tagged with `[methodology]` only — these are domain-nonspecific and not feedback targets (design document Section 4-2).

### 2. Routing

Route each collected learning along two independent axes (design document Section 4-1):

**Axis 1 — Subject area** (determines target file):

| Subject Area | Target File |
|---|---|
| Terminology / concept definitions | `concepts.md` |
| Scope / sub-areas | `domain_scope.md` |
| Logic rules | `logic_rules.md` |
| Structural rules | `structure_spec.md` |
| Dependency rules | `dependency_rules.md` |
| Questions / queries | `competency_qs.md` |
| Extension scenarios | `extension_cases.md` |
| Conciseness rules | `conciseness_rules.md` |

**Axis 2 — Change type** (CRUD):

| Type | Description |
|---|---|
| `add` | New item addition |
| `modify` | Existing item modification |
| `remove` | Existing item removal suggestion |

1:N routing is allowed — one learning may affect multiple files. The two axes are independent.

### 3. Compare

For each (learning, target file) pair, perform semantic comparison (design document Section 4-3):

| Result | Definition |
|---|---|
| `duplicate` | Removing the existing item would fully preserve the learning's information |
| `complement` | The learning contains new information absent from the existing item |
| `conflict` | The learning and existing item make incompatible claims about the same subject |

The three states are mutually exclusive. Judgment is LLM-based with rationale included in output.

For 1:N routed learnings, compare is performed independently per target file (design document Section 4-3).

`duplicate` items are excluded from further processing.

### 4. Suggestion Generation

For `complement` and `conflict` items, generate concrete change suggestions in diff format showing what would be added, modified, or removed in the target file.

### 5. User Presentation

Present suggestions grouped by target file.

- `complement` items: show the proposed addition/modification.
- `conflict` items: show both sides (current document content vs. learning content) and the consequences of each choice.

### 6. User Approval

Per-item approval. Partial approval is supported — the user may approve some suggestions and reject others, including approving for one target file while rejecting for another in 1:N routing cases.

### 7. Application and SEED Marker Handling

Apply approved changes to the domain document files.

**SEED marker handling** (design document Section 4-7, step 8): When applied content replaces content within a SEED marker block, ask the user whether to remove the SEED marker. The agent does not remove SEED markers automatically — removal requires explicit user approval.

### 8. Feedback Log

Record the feedback session in `feedback-log.md` at the domain's location (design document Section 4-5).

Log format and fields follow the specification in design document Section 4-5.

# onto Work Item Canonical Output (2026-04-13)

본 문서는 onto 축 A/B/C/D work item 의 **canonical output seat** 이다. M-04 Phase A 에서 schema + seat 구조만 확정하고, 실제 work item 은 **M-06 (축별 Work Item 초안)** 에서 작성된다.

## 0. 위상·경계

- **canonicality**: canonical-advancing (축 A/B/C/D 구현의 "무엇을 할지" 를 canonical 로 정렬하는 output seat).
- **authority_stance**: non-authoritative-design-surface. 본 파일은 design record 이며 rank 1 authority 아님. authority SSOT 는 `authority/core-lexicon.yaml` (enum refs) 및 `development-records/design/20260413-onto-direction.md` (§1 정본).
- **소비 제약**: 본 파일의 W-ID 인용은 **M-06 이후** 에만 허용. M-04 Phase A 단계의 W-ID 는 schema 검증용 dry-run 전용.

## 1. Schema 명세 (v1.4 기준, 18 필드)

v5.1 meta 표의 필드 집합을 기준으로 v1 초판을 확정하고, v1.4 에서 `compound_member` (18번째 필드) 를 additive 로 추가한다. 필드 수는 meta v5.1 원문(활동 포함 17개) 기준이며, 일부 초기 문건에서 "15 필드" 라고 표기된 것은 기점 버전 차이(v4=11, v5=16, v5.1=17)로 인한 표기 혼선 — 본 schema 가 canonical. **v1.4 시점 필드 수 = 18** (compound_member 추가).

### 1.3 필드 간 관계·불변식

- **id ⇔ axis**: `id` 의 axis token 이 `activity` 와 충돌하지 않아야 한다 — 단, activity-축 관계는 soft (A0=scope-runtime 같은 infra advance 는 `axis=B` + `activity=reconstruct` 조합이 자연스러울 수 있음). validator 경고 수준 (M-07 hard reject 아님).
- **depends_on ⇒ acyclic**: W-ID 간 의존은 DAG. cycle 발견 시 M-07 reject.
- **depends_on 의미 구분** (C-04 해소, v1.1 추가):
  - `W-[ABCD]-NN` 원소 = **실행 DAG 선행 관계** (해당 work item 이 선행되어야 본 work item 이 실행 가능)
  - `BL-NNN` 원소 = **resolves 관계** (본 work item 이 해당 backlog 항목을 해결)
  - `DL-NNN` 원소 = **resolution_stage 도달** (본 work item 의 완료가 해당 deferred item 을 resolved 로 전환)
  - 세 종류는 의미가 다르므로 M-05 dep graph 는 각각 별도 edge 로 모델링.
- **provenance vs depends_on 분리**: §1 정본 anchor / DR-NN / URL 은 **provenance 필드 전용** (origin 증빙). `depends_on` 은 W/BL/DL ID 만 허용 (실행·해결 관계).
- **provenance 필수**: 최소 1개. origin 이 없으면 본 M-03 gap 도출 근거가 없음을 의미 (reject).
- **activity 복수 귀속**: 활동 간 interface 성격인 work item (예: `[review, learn]` — review 실행 중 learning 자동 수집) 만 복수 허용. 기본 단일.
- **lifecycle_status 전이**: `active ↔ deferred` 양방향, `active → done`, `active → deprecated → archived` 일방향. 전이 주체는 M-08 protocol.
- **Migration Contract**: axis 변경 시 `previous_id` + `deprecated_aliases` 기록 필수. rename log 는 `rewrite_trace` 에 `{date, change_type: "axis_move", old_id: "W-A-05", new_id: "W-B-12"}` 형식.
- **Cluster 판정 알고리즘** (C-05 해소, v1.1 추가, M-07 책임):
  - 두 work item A, B 가 **동일 cluster 후보** 인 조건: A.files 와 B.files 의 glob expansion 결과가 1개 이상의 파일을 공유하거나, 한 glob 의 prefix 가 다른 glob 의 expansion 에 포함.
  - Cluster 내 work item 은 순차 실행 필수 (같은 파일 동시 수정 금지).
  - 예외: `files` 가 빈 list 인 work item 은 cluster 판정 대상이 아님 (ownership_boundary 만으로 판정).
- **ownership_boundary ↔ files prefix 불변식** (C-05 해소): 각 work item 의 `files` glob 에서 first `**` 또는 `*` 이전 prefix 는 `ownership_boundary` 에 포함되어야 한다 (role-based boundary 는 예외 — 해당 role 의 책임 범위 내 glob 허용).
- **Compound 무결성** (v1.4 추가, DR-M04-03): `compound_member` 가 non-null 인 W-ID 집합에 대해 다음 3 불변식 필수:
  - (i) 같은 `compound_member.id` 를 공유하는 W-ID 들의 `ordinal` 값은 `1..total` 로 **연속·중복 없음**. gap 또는 중복 시 schema validator reject.
  - (ii) 각 W-ID 의 `depends_on` 에 **직전 ordinal** 의 동일 compound 멤버 W-ID 가 포함되어야 함 (ordinal=1 은 예외). 직전 phase 미포함 시 reject.
  - (iii) 같은 `compound_member.id` 공유 W-ID 들의 `total` 값은 **일관**. 값이 다르면 reject.
  - (iv) Compound 멤버 W-ID 는 `rewrite_trace` 에 최초 entry 로 `{change_type: "compound_expansion", source_bl: "BL-NNN" 또는 null, compound_id: "<id>", phase: <ordinal>}` 을 포함해야 함 (M-06 분해 시점 기록).
  - 본 불변식은 M-06 work item 작성 + M-07 cluster 판정 시 모두 검증. reject 시 M-06 내 patch 또는 M-04 reopen.

### 1.5 schema version 진화 규칙

- **표기법** (G-3 / C-10 해소, v1.1 명시): `schema_version: v{major}.{minor}` (예: v1, v1.1, v2). minor 는 additive 시 증가, major 는 breaking 시 증가. integer-only 는 v1, v2 처럼 minor 0 표기 생략 가능.
- **additive change** (필드 추가, enum 값 추가, constraint 완화): `schema_version` minor bump. enum 값 추가 시 `lexicon_version` 도 minor bump (core-lexicon SSOT 동기화).
- **breaking change** (필드 삭제, enum 값 삭제, 의미 변경, regex 자릿수 축소): `schema_version` major bump. 기존 item 전수 migration (Expansion Protocol Migration Contract). lexicon_version 도 의미 변경 시 major bump (core-lexicon `schema_version_evolution` 규약 준수).
- **ID 공간 확장 trigger** (C-03 해소, v1.1 추가): `axis` 별 `nn:02d` ID 가 99 에 도달하면 `schema_version` major bump + ID regex 를 `^W-[ABCD]-\d{2,3}$` 또는 `\d{3}` 으로 확장. **선제적 확장** 도 옵션 — Principal 판단으로 본 schema v1.x 단계에 미리 `\d{2,3}` 으로 확장 가능 (item 0건 상태에서 확장 비용 최저).
- **archived ID nn 재사용 금지** (Migration Contract 보강): 본 schema 는 W-ID 를 stable 로 가정. archived 로 이관된 ID 의 nn 을 새 work item 에 재할당 금지 — 새 item 은 axis 내 가장 큰 nn+1 로 reissue.
- **trigger**: M-04 Phase A 결과 schema 변경 필요 시 M-04 reopen. M-06 이후 발견 시 M-08 refresh protocol 경유.
- **downstream reader 재파싱 요건**: DL-027 (v2-review E-7) 에서 수렴. schema_version 변경 시 본 파일 frontmatter + consolidated frontmatter + 각 work item 의 schema 준수 재검증.

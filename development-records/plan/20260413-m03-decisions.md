## DR-M03-03: Canonicality 3분류 판정 규칙

- **status**: applied
- **origin_ref**: v5.1 meta task 정의 (M-03 Canonicality Gate) + onto-direction §1.0 세 측면 + §1.2 5 활동 + §1.5 4 축
- **default_rule**:

  - **supporting**: canonical 작업을 지원하는 인프라·도구·정책·학습 소비·lexicon lifecycle 등. 대부분의 axis B (infrastructure) 항목, domain asset, process rule, diagnostic contract 등.
  - **scaffolding**: spec fix, post-PR cleanup, rename, legacy migration, cross-ref 교정, 잔여 결함 처리. 직접 canonical advance 아니나 product hygiene 을 위해 수행.
- **subordinate_rules** (C-C review 반영):
  - **(i) Governance decision execution → canonical-advancing**: §1.0~§1.5 에 명시된 governance decision (폐기·보류·다음 작업·축 선행 등) 의 execution 산출도 canonical-advancing 으로 간주. 즉 decision modality 가 §1 정본에 직접 기록된 경우, 그 decision 의 첫 실행(폐기 집행·승격 기준 수립·선행 확인) 은 "직접 advance" 에 포함. **적용 사례**: BL-121 (§1.2 "폐기된 개념: ask" 직접 집행).
  - **(ii) Seat 입력 자료 정돈 (seed 정돈) → supporting**: 축·활동 seat 건설 자체가 아니라 그 seat 의 입력·신호·canonicality 결정의 선결 정돈은 supporting. seat 건설 (설계·구현 주체) 과 구분. **적용 사례**: BL-088 (event_marker signal 의미 정돈 — C autonomy seat 의 입력 정돈), BL-089 (creation_gate dead code 제거 — scaffolding 으로 분류; seed 정돈 아님), BL-090 (health-history canonical 결정 — supporting; canonicality decision 자체가 supporting seat).

(2) 같은 §1 조항에 의존한 canonical-advancing 이 여러 BL 에 분산되어 cross-ref 관계가 복잡해질 때 → M-05 Pre-draft Dependency Modeling 에서 `precedes/depends_on` 필드로 구조화. (3) **§1 정본 개정 이벤트** — `onto_direction_ref` snapshot anchor 와 현재 §1 문서 해시·as_of 가 불일치하면 canonical-advancing 8건 전수 재판정 필요 (Principal 개입).
- **onto_direction_ref**: `development-records/design/20260413-onto-direction.md#as_of=2026-04-13` (canonical-advancing 판정 snapshot anchor — 본 DR 은 이 snapshot 에 bound). §1 개정 시 본 ref 갱신 + 8건 전수 재검토.

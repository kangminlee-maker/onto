## DR-M05-01: 축 순서 재정립 + 축 간 의존 (D0→B→A→C + D continuing)

- **status**: applied
- **origin_ref**: v5.1 meta M-05 spec "축 순서 재정립" + §1 정본 §1.5 축 간 의존
- **context**: §1 정본 §1.5 은 축 간 의존을 "A→B / C→{A,B} / D→{A,B,C}" 로 선언했으나, 이는 **선행 관계 선언** 이며 시간 축 실행 순서는 아직 풀어지지 않았다. v5.1 meta 는 이를 "D0 bootstrap → B → A → C + D continuing (cross-cutting lane)" 으로 시간 축으로 풀었다. 본 DR 은 이 풀림을 M-05 dep graph 의 **축 backbone** 으로 applied 확정한다.
- **Decision**: 축 실행 순서는 `D0 → B → A → C + D continuing`. 축 D 는 **완료되는 선행이 아닌 cross-cutting lane** 임을 명시.
- **default_rule**:
  - **D0 (bootstrap)**: 축 D 의 최소 성문화. BL-123 중 (a) lexicon provisional lifecycle 성문화 (seed → candidate → provisional → promoted 전이 규칙 + `lifecycle_status` 관리 위치), (b) 인용 금지 검증 도구. **완료 전 B/A/C 진입 금지**.
  - **B (기반 인프라)**: D0 이후 착수. A 의 선행. 주요 내용: scope-runtime (A0 framework 추출 포함), readers, middleware, data-seats.
  - **A (entrypoint 구현)**: B 완료 후. 5 활동 (review / design / reconstruct / learn / govern) command·process·calling convention.
  - **C (자율성 진화)**: A·B 최소 1사이클 운영 후 설계. drift 판정 engine + 큐·승인 + 수준 0→1→2.
  - **D continuing**: D0 이후 cross-cutting lane. 각 축 work item 이 새 lexicon term 등장 시 등록·관리. 별도 W-ID 없이 lifecycle 자체로 운영. M-08 refresh 가 주기적 scan.
  - **BL-123 내부 분할**: D0 = lifecycle 성문화 + 인용 금지 도구 (W-D-NN). D continuing = 각 축 등록 활동 (별도 W-ID 없음).
  - **BL-122 처리**: knowledge → principle 승격 경로 (§1.2 "보류 중") = D continuing 의 reactivation trigger — govern runtime (DL-021 M-06 신규 도출) 완료 후 재활성.
- **escalation_trigger**:
  - (1) §1 정본 §1.5 개정 시 본 DR 재실행 + dep graph §1 갱신.
  - (2) D0 범위 확장 필요 (예: BL-123 외 다른 lexicon 관련 work 가 D0 에 포함) 시 DR-M05-NN 추가 또는 본 DR revise.
  - (3) 축 A·B 1사이클 운영 종료 시점 정의 — M-06 종료가 "1사이클" 으로 간주되는지는 M-07 lifecycle balance check (DL-023) 에서 판정.
  - (4) D continuing 이 별도 work item 화 필요 (누적 term 수 과도) → M-08 refresh 에서 판정.
- **owning_artifact**: `development-records/design/20260413-onto-todo-dep-graph.md` §1 / §2.
- **consumers**: M-06 (축별 work item 분배 — D0/B/A/C 순서 준수), M-07 (cluster 판정 시 축 sequence 검증), M-08 refresh (D continuing scan).
- **applied_cases**:
  - onto-todo-dep-graph.md §1 (축 의존 그래프) + §2 (D bootstrap vs continuing) 에 반영.
  - M-06 은 W-D-NN (D0 outcomes, BL-123 일부) 를 선행 work item 으로 배치해야 함.

## DR-M05-02: Compound Sequencing 표기 규약 (DL-015 해소)

- **status**: applied
- **origin_ref**: DL-015 (v1-review C-8 MODERATE — Compound/Phase sequencing 구조 승격)
- **context**: consolidated v3.2 에는 다수 row 가 단일 backlog entry 이지만 실제로는 strict sequence 로 연결된 N 개의 work item 묶음이다. 예: agent_id_rename Phase 0→5 (BL-104~109), project_build 3차(§2) → 8차(§1) 순서, Business Wave 1-3 (BL-101~103). 이를 `item_granularity: consolidation-row (not work-item)` 으로 선언하되, M-06 work item 차원에서는 N W-ID 로 분해하는 표기 규약이 필요.
- **Decision**: Compound = 2 이상의 W-ID 가 strict sequence 로 연결된 work item 묶음. `compound_id` metadata + `depends_on` 체인 + Notes compound_member 3종 표기로 규약화.
- **default_rule**:
  - **Compound 식별**: onto-todo.md v1.x work item 의 Notes 또는 frontmatter 확장 필드 `compound_id: <snake_case_name>`.
  - **Sequencing**: 각 W-ID 의 `depends_on` 에 직전 Phase W-ID 기록. `depends_on` 3종 의미 구분 (onto-todo.md §1.3, C-04) 중 **W-[ABCD]-NN 원소 = 실행 DAG 선행** 해당.
  - **Notes**: 각 W-ID 의 Notes 에 `compound_member: <compound_id> phase_<n> of <N>` 기록.
  - **Granularity 전환**: consolidated v3.x row 1건 → onto-todo.md N W-ID (M-06 책임). consolidated frontmatter 에 `item_granularity: consolidation-row` (현재 암묵적) 명시 필요 시 M-00 재revision 에서 처리.
  - **rewrite_trace**: compound 분해 시 각 W-ID 의 `rewrite_trace` 에 `{change_type: "compound_expansion", source_bl: "BL-NNN", compound_id: "<name>", phase: <n>}` 기록. 현 schema v1.2 의 `change_type` enum 에 `compound_expansion` 이 **미포함** — M-06 에서 처음 사용 시 schema v1.3 minor bump 필요 (additive).
  - **Pre-identified compound 3 사례** (M-06 진입 시 최소 cover):
    - `agent_id_rename` (BL-104~109, 6 Phase)
    - `build_review_cycle` (§2 build_3rd → §1 build_8th, 2 단계)
    - `business_domain_wave` (BL-101~103, 3 단계; BL-101 이 Wave 1/2/3 로 nested compound)
- **escalation_trigger**:
  - (1) M-06 에서 pre-identified 3 사례 외 새 compound 발견 시 dep graph §5.3 에 추가 + 본 DR applied_cases 업데이트.
  - (2) `compound_expansion` change_type 사용 시 onto-todo.md schema v1.3 minor bump (M-04 minor reopen — 단일 DR-M04-NN 추가 수준).
  - (3) compound 내 Phase 간 의존이 strict sequence 가 아닌 경우 (예: 병렬 가능한 sub-phase) → compound 정의 재검토. M-07 cluster 판정에서 조기 발견 목표.
- **owning_artifact**: `development-records/design/20260413-onto-todo-dep-graph.md` §5 (Compound Sequencing 규약).
- **consumers**: M-06 (compound row 를 N W-ID 로 분해), M-07 (compound 같은 cluster 순차 실행 후보 판정), onto-todo.md schema v1.3 (필요 시 `compound_expansion` enum 추가).
- **applied_cases**:
  - dep graph §5.3 에 3 compound pre-identified (agent_id_rename, build_review_cycle, business_domain_wave).
  - M-06 실행 시 본 규약 준수하여 compound row 를 N W-ID 로 분해.

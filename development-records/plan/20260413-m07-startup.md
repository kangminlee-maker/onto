## 2. M-07 정의 (v5.1 meta 기준)

### 목적

M-06 산출 (onto-todo.md v2, 139 W-ID) 의 **구조적 정합성 검증** + **병렬 실행 가능 cluster 판정** + **생애주기 분포 균형 점검**. M-08 Refresh Protocol 의 입력.

### Scope

| 다루는 것 | 다루지 않는 것 |
|---|---|
| 전체 W-W DAG acyclic 검증 | 신규 W-ID 생성 (M-06 책임) |
| Cluster 판정 (files glob 공유 검출 → 순차 실행 지정) | 실제 work item 실행 (구현 단계) |
| Compound 무결성 불변식 4건 × 3 compound 재검증 | Schema 변경 (M-04 책임) |
| depends_on 의미 구분 (W/BL/DL) 정합성 검증 | 축 backbone 재정의 (M-05 책임) |
| Lifecycle balance check (DL-023) — 축 × 활동 × lifecycle 분포 | 개별 W-ID 내용 수정 (M-06 reopen) |
| Consistency check (DL-024) — 3지점 cross-count 검증 | Refresh protocol 수립 (M-08) |
| Revise escalation 판정 (defect 원인 stage 식별) | — |

### 검증 항목 5가지

1. **DAG acyclic**: depends_on 의 W-ID edge 를 전수 순회. cycle 발견 → reject + M-06 reopen.
2. **Cluster 판정**: 두 W-ID 의 `files` glob expansion 이 1+ 파일 공유 → 같은 cluster → 순차 실행. files 빈 W-ID 는 제외 (ownership_boundary 만으로 판정).
3. **Compound 무결성 재검증**: (i) ordinal 연속·중복 없음, (ii) 직전 phase depends_on 포함, (iii) total 일관, (iv) 최초 rewrite_trace compound_expansion 기록.
4. **Lifecycle balance**: 축 × 활동 × lifecycle_status 교차 분포. canonical-advancing / deferred 편향 점검. `scaffolding_ratio_trend` health metric (DL-023).
5. **Consistency check 3지점** (DL-024): (a) M-00 entry count(123) ≥ M-03 gap count(101), (b) M-03 gap count(101) ≤ W-ID count(139, 신규 DL 포함), (c) 각 W-ID 18 필드 전수 채움 확인.

- 원인 = M-06 (W-ID authoring 결함) → M-06 reopen

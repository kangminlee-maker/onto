**Wire 작업의 약속 행동**:
1. boot 시 `.onto/config.yml` read → `loadReconstructDogfoodSwitches()` 호출
2. `checkSwitchInvariants()` 결과 ok=false 시 `phase3_rationale_review_requires_v1_inference` / `write_intent_inference_to_raw_yml_requires_v1_inference` halt
3. Hook α/γ/δ 분기를 switch 로 gate:
   - `v1_inference.enabled == false` → Hook α/γ/δ 모두 skip (`alpha_skipped` / `gamma_skipped` 직행)
   - `phase3_rationale_review.enabled == false` → Hook γ 만 skip (Hook α/δ 는 활성)
   - `write_intent_inference_to_raw_yml.enabled == false` → raw.yml save 시 element-level `intent_inference` block omit
4. boot 시 부재 config detect 시 console.warn (또는 session-log) 에 `reconstruct_config_absent_default_v1_applied` emit (post-PR232 backlog A2 동시 처리)

### 2.2 Wire path — 두 후보 (다음 세션 결정)

#### Path (A): prompt body 에 switch read 지시 추가 (LLM-Native)

- 변경: `.onto/commands/reconstruct.md` 에 boot 단계에 `.onto/config.yml` 의 `reconstruct:` block read + switch 분기 LLM 지시 추가
- 장점: onto 의 LLM-Native principle 과 일관 (Phase 0 ~ Phase 4 가 LLM agent orchestration)
- 단점: deterministic 보장 어려움 — race / atomic / fsync 가 LLM 책임 밖. invariant rejection 의 *halt* 가 LLM 의 self-discipline 의존
- 적합: *helper-only layer* 의 자연 진화. 본 PR scope 와 일관

#### Path (B): runtime Coordinator class 신설 (deterministic)

- 변경: `src/core-runtime/reconstruct/coordinator.ts` 신설 — `runReconstructWire(args, deps)` 함수 export
  - boot 시 `loadReconstructDogfoodSwitches()` + `checkSwitchInvariants()`
  - Hook α/γ/δ caller 들이 본 함수를 통과하면서 switch 분기
  - `cli/reconstruct-entry.ts` (또는 기존 entry) 가 본 함수를 호출
- 변경: hook-alpha.ts / hook-gamma.ts / hook-delta.ts 의 caller path 가 coordinator 를 거치도록 wrapping (hook 본문 변경 0 — caller 만 변경)
- 장점: deterministic — atomic / fsync / invariant rejection 의 runtime 보장
- 단점: scope 큼 (~수백 lines), 본 PR 의 atomic commit pattern 변경. e2e-smoke.test.ts 의 mock dispatcher 패턴이 *real Coordinator* 호출로 일부 변환 필요

### 2.3 Path 결정 criteria

| 기준 | Path A 우선 | Path B 우선 |
|---|---|---|
| onto 의 LLM-Native principle 일관성 | ✓ | — |
| atomic / fsync / race 보장 | — | ✓ |
| 본 PR 의 atomic commit pattern 보존 | ✓ | — |
| dependency invariant rejection (runtime halt) 보장 | — (LLM self-discipline) | ✓ |
| 다른 활동 (review / evolve / learn / govern) 의 wire 패턴과 일관 | review 가 deterministic Coordinator (review-invoke.ts) — Path B | review 가 deterministic — Path B |
| 본 PR scope 와 자연 | ✓ helper layer 자연 진화 | scope 확대 |

**판단**: review 활동이 이미 `review-invoke.ts` 로 *deterministic Coordinator* 패턴을 사용하므로 *5 활동 일관성* 측면에서 Path B 가 자연. Path A 는 *부분 wire* (prompt 안 switch read) 만, *invariant rejection halt* 가 부재라 SDK-like invariant 보장 약함. **Path B 권고**.

## 3. A3 — principal_provided_rationale persistence 결정 + patch

### 3.1 Finding 본질 (PR #232 r5 C3 BLOCKING)

**Spec (Step 4 §3.6.1 canonical)**: `elements[].intent_inference.principal_provided_rationale` 가 element-level single seat. raw.yml 저장 시 §4.3 element-level intent_inference block 에 포함.

**Implementation 현재 (`phase-3-5-runtime.ts` line 329~358 `applyToElement`)**:
```typescript
if (decision.action === "modify" || ... === "provide_rationale" || ... === "override") {
  const ppr = decision.principal_provided_rationale!;
  next.inferred_meaning = ppr.inferred_meaning;       // overwrite
  next.justification = ppr.justification;              // overwrite
```

## 5. 위험 + 주의사항

### 5.1 A1 wire 의 회귀 위험

- 본 PR 의 helper layer + e2e-smoke test 가 wire 의 *spec contract* 정의 — wire 가 contract 를 따르면 회귀 0 기대
- 다만 *real coordinator* 가 mock dispatcher 와 다른 fail mode 가질 수 있음 (timeout / network / fsync race) — 본 wire commit 은 *runtime path* 만 wire, *production LLM call* 은 별도 (mock 유지)

### 5.2 A3 path (a) 의 backward compatibility

- `IntentInference` 에 optional field 추가 — schema 변경이지만 기존 raw.yml read 시 `undefined` 로 자연 처리
- 기존 e2e-smoke.test.ts Cycle 4 assertion 이 `inferred_meaning overwrite` 검증 — patch 후 *principal_provided_rationale persist* 검증으로 정정 필요
- *과거 raw.yml* (Cycle 4 assert 후 정정 전) 의 element 는 `principal_provided_rationale` field 부재 — read 시 `undefined` 자연 처리, 회귀 없음

### 5.3 A1 path 결정 reversal 가능성

- Path B (Coordinator) 채택 후 LLM-Native principle 위반 우려 시 *부분 reversion* 가능 — coordinator 가 LLM dispatch 하는 형태 (deterministic boot + LLM orchestrate) 로 hybrid 가능
- 즉 Path B 가 *strict deterministic* 이지만 hybrid 가능성 보존

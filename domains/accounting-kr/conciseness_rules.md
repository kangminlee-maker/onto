# accounting-korea Domain — 간결성 규칙

이 문서는 conciseness가 간결성 검증 시 참조하는 한국 회계 도메인 특화 규칙이다.
순서: **유형(허용/제거) → 검증 기준 → 역할 경계 → 측정 방법**.

---

## 1. 허용되는 중복

각 규칙에 심각도 태그 부여:
- **[MUST-ALLOW]**: 제거 시 시스템이 깨지는 중복. 반드시 유지.
- **[MAY-ALLOW]**: 편의를 위해 유지하는 중복. 통합 가능하나, 이점이 명확할 때만 제거.

### 회계-세무 이중 추적

- [MUST-ALLOW] K-IFRS/K-GAAP 회계처리와 법인세법상 처리의 병행 기록 — 동일 거래가 회계와 세무에서 다르게 처리될 수 있다. 세무조정(익금산입/손금불산입/손금산입/익금불산입)은 이 차이를 관리하는 절차이며, 하나로 통합하면 재무보고와 세무 신고 양쪽 모두 불가. 근거: 법인세법 제14조(각 사업연도의 소득 계산)
- [MUST-ALLOW] 장부금액과 세무기준액의 이중 추적 — 이연법인세 산출을 위해 동일 자산·부채에 대해 두 금액을 동시에 유지. 한쪽 제거 시 이연법인세 계산 불가. 근거: K-IFRS 1012호(법인세)

### 증빙 체계 중복

- [MUST-ALLOW] 세금계산서상 금액과 회계 분개 금액의 병행 — 세금계산서는 부가세법상 증빙(공급시기 기준 발행), 분개는 회계 기록(수익인식 시점 기준). 발행 시점·인식 시점 차이로 금액 불일치 가능. 양자 모두 필요
- [MUST-ALLOW] 증빙 유형별 병행 존재(세금계산서·카드매출전표·현금영수증) — 각 증빙이 별도의 법적 기능(부가세 증빙, 소득공제, 매입세액 공제)을 수행. 단, 동일 거래에 대해 세금계산서와 카드매출전표를 중복 발행하는 것은 이중 계상 리스크 (→ 제거 대상 '증빙 중복' 참조)

### 재무제표 간 교차 참조

- [MUST-ALLOW] 재무제표 간 연결 항목의 양측 기록 — 당기순이익이 포괄손익계산서와 자본변동표 양쪽에 표시. 현금 변동이 현금흐름표와 재무상태표 양쪽에 표시. 재무제표 간 무결성 검증 목적. 근거: K-IFRS 1001호(재무제표 표시) — 4개 연결 관계는 dependency_rules.md '재무제표 간 연결' 참조
- [MAY-ALLOW] 주석에서 본문 수치 재기술 — 본문 항목의 세부 내역을 주석에서 설명. 합계 일치 시 유지, 단순 반복이면 제거 대상. 근거: K-IFRS 1001호 문단 112~116(주석의 구조)

### K-IFRS/K-GAAP 이원 체계

- [MUST-ALLOW] 동일 거래에 대한 K-IFRS 처리와 K-GAAP 처리의 병행 *설명* — 전환 기업(비상장→상장 시 K-GAAP→K-IFRS 전환 필요), 비교 목적, 또는 두 기준 적용 기업이 공존하는 그룹에서 필요. 양 기준의 차이를 명시하는 것은 혼용 방지에 필수. 주의: 이것은 "설명"의 병행이지 "적용"의 병행이 아님 — 동일 기업에 두 기준 혼용 적용은 MUST-REMOVE (→ 제거 대상 '기준 적용 중복' 참조)

## 2. 제거 대상 패턴

각 규칙에 심각도 태그 부여:
- **[MUST-REMOVE]**: 존재 자체가 오류나 잘못된 추론을 유발하는 중복.
- **[SHOULD-REMOVE]**: 크게 해롭지 않으나 불필요한 복잡성을 추가하는 중복.

### 기준 적용 중복

- [MUST-REMOVE] 동일 기업에 K-IFRS와 K-GAAP 기준 혼용 적용 — 하나의 기업은 하나의 기준만 적용. 상장기업·금융기관은 K-IFRS 의무, 비상장 외부감사 대상은 K-GAAP(K-IFRS 선택 가능). 혼용은 외부감사법 위반. 근거: 주식회사 등의 외부감사에 관한 법률 제5조
- [SHOULD-REMOVE] K-IFRS 개념체계가 이미 요구하는 원칙을 개별 회계정책에서 재선언 — 기준 개정 시 갱신 누락 리스크

### 세무 중복

- [MUST-REMOVE] 동일 거래에 대한 이중 세무조정 — 한 거래를 두 번 조정하면 과세소득 왜곡. 세무조정은 익금산입/손금불산입(가산)과 손금산입/익금불산입(감산)의 4유형이며, 동일 항목에 중복 적용 시 신고 오류
- [SHOULD-REMOVE] 동일 세무조정 근거를 여러 조정명세서에서 반복 기술 — 근거는 하나의 명세서에 기재하고 나머지에서 참조

### 증빙 중복

- [SHOULD-REMOVE] 동일 거래에 세금계산서와 카드매출전표를 모두 발행 — 이중 증빙은 매출 이중 계상 리스크. 하나의 거래에 하나의 주요 증빙. 부가가치세법 제36조 제3항: 신용카드매출전표 등을 교부한 경우 세금계산서를 별도 발급하지 않음. 이중 발행 시 부가세 신고에서 중복 금액 차감 처리 필요

## 3. 최소 세분화 기준

하위 분류는 다음 중 **최소 1개**를 충족해야 허용. 충족하지 않으면 상위와 합산.

1. **역량질문 차이**: competency_qs.md의 질문에 대해 다른 답을 생성하는가?
2. **제약 차이**: 다른 인식·측정·공시 요구사항이 적용되는가?
3. **의존 차이**: 다른 기준/법령에 의존하거나, 다른 재무제표/신고서 항목에 연결되는가?

예시:
- 법인세 세무조정과 부가세 신고는 적용 법률(법인세법 vs 부가가치세법)·신고 주기(연 1회 vs 연 4회)·과세 기준(소득 vs 공급가액)이 모두 다르므로 별도 분류 정당
- 매출세액과 매입세액은 동일 부가세법 내지만 공제 규칙과 검증 질문이 다르므로 별도 분류 정당
- 전자세금계산서와 수정세금계산서가 별도 개념이면 합산 검토 대상 (동일 증빙 체계, 수정은 원본의 변형)

## 4. 경계 — 도메인 특화 적용 사례

경계 정의의 정본은 `roles/conciseness.md`. 이 섹션은 한국 회계 도메인 특화 적용 사례만 기술.

### pragmatics 경계

- conciseness: 불필요한 요소가 **존재하는가?** (구조 수준)
- pragmatics: 불필요한 정보가 질의 실행을 **방해하는가?** (실행 수준)
- 예: 미사용 세무조정 항목 존재 → conciseness. 과다한 조정명세서가 핵심 조정 식별 방해 → pragmatics

### coverage 경계

- conciseness: 없어야 할 것이 존재하는가? (축소 방향)
- coverage: 있어야 할 것이 빠져 있는가? (확장 방향)
- 예: 원천징수 관련 처리 누락 → coverage. 동일 원천징수를 여러 경로로 중복 처리 → conciseness

### logic 경계 (선후 관계)

- logic 선행: 논리적 동치 판단 (함의 관계)
- conciseness 후행: 동치 확인 후 제거 여부 결정
- 예: K-IFRS 개념체계의 발생기준 원칙이 개별 회계정책의 기간 귀속 규칙을 함의 → logic가 동치 판단 → conciseness가 "개별 정책 재선언 불필요" 판단

### semantics 경계 (선후 관계)

- semantics 선행: 의미적 관계 판단 (동의어·포함관계 여부)
- conciseness 후행: 관계 유형에 따라 통합 또는 관계 명시 결정
- 예: 손익계산서/포괄손익계산서는 포함관계(⊃)이지 동의어가 아니다 → semantics가 관계 유형 판단 → conciseness가 '통일이 아닌 관계 명시' 판단

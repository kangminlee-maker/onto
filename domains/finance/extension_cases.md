# Finance Domain — 확장 시나리오

## 분류 축

| 축 | 값 | 설명 |
|---|---|---|
| 변경 유형 | 노드 추가 / 관계 추가 / 규칙 변경 / 원천 변경 | 온톨로지에 미치는 영향 |
| 영향 범위 | 국소(단일 노드) / 광역(복수 규칙) | 변경 파급 범위 |
| 난이도 | 낮음 / 중간 / 높음 | 구현 복잡도 |

## 시나리오 1: 새 회사 추가

- **상황:** 기존에 없던 기업(예: 카카오뱅크)의 재무제표를 온톨로지에 추가
- **변경 유형:** 노드 추가
- **영향 범위:** 국소
- **난이도:** 낮음
- **필요 작업:**
  1. Entity 노드 생성 (기업 코드, 업종 분류)
  2. 해당 기업의 FinancialFact, Period, Note 노드 생성
  3. 기업별 확장 계정(ENTITY_EXTENSION)의 L2 매핑 수행
  4. 참조 무결성(RI01, RI02) 검증
- **주의사항:** 금융업종 기업인 경우 XBRL role 코드 체계가 상이하므로 파서 설정 변경 필요

> **학습 반영 (onto_coverage):** ENTITY_EXTENSION 비율이 84.8%에 달하므로, 새 기업 추가 시 대부분의 계정과목이 확장 계정이다. L2 매핑 없이는 기업 간 비교(Q05~Q07)가 불가능하다.

## 시나리오 2: 분기보고서 추가

- **상황:** 연간 보고서만 존재하던 기업에 분기보고서 추가
- **변경 유형:** 노드 추가
- **영향 범위:** 국소
- **난이도:** 낮음
- **필요 작업:**
  1. 분기별 Period 노드 생성 (Q1: 01-01~03-31, Q2: 01-01~06-30, Q3: 01-01~09-30)
  2. 분기별 FinancialFact 노드 생성
  3. 기간 규칙(PR03) 적용 — 분기 수치와 연간 수치의 직접 비교 방지
  4. 누적 기간 vs 단일 분기 구분 주의 (Q2 = 상반기 누적 or 2분기 단독)
- **주의사항:** 분기보고서는 감사 대신 검토를 받으므로 신뢰도 수준이 다름

## 시나리오 3: 택소노미 변경

- **상황:** IFRS 택소노미 버전 업데이트 (예: 2023→2024)
- **변경 유형:** 규칙 변경
- **영향 범위:** 광역
- **난이도:** 높음
- **필요 작업:**
  1. 변경된 Concept 노드 식별 (추가/삭제/이름변경)
  2. HAS_CONCEPT 관계의 참조 무결성(RI01) 재검증
  3. 동의어 매핑(concepts.md) 갱신
  4. 기존 FinancialFact의 Concept 참조 업데이트 또는 버전 병행 유지
- **주의사항:** 택소노미 변경은 Concept 계층에만 영향하고, 인스턴스(FinancialFact)는 독립적으로 관리

> **학습 반영 (onto_coverage):** 택소노미/인스턴스 분리 전략에 따라, 택소노미 변경 시 인스턴스를 일괄 갱신하지 않고 매핑 레이어(L2)에서 버전 간 대응표를 유지한다. 이 방식은 과거 데이터의 안정성을 보장한다.

## 시나리오 4: 비재무 데이터 통합

- **상황:** ESG 데이터, 인력 현황 등 비재무 정보를 온톨로지에 추가
- **변경 유형:** 노드 추가 + 관계 추가
- **영향 범위:** 광역
- **난이도:** 중간
- **필요 작업:**
  1. 새 노드 유형 정의 (예: NonFinancialFact, ESGMetric)
  2. Entity와의 관계 정의 (REPORTED_BY 재사용 가능)
  3. Period 관계 정의 (IN_PERIOD 재사용 가능)
  4. 기존 회계 항등식(EQ01~EQ03)에는 영향 없음 확인
  5. 비재무 데이터 전용 논리 규칙 별도 정의
- **주의사항:** 비재무 데이터는 XBRL로 제공되지 않는 경우가 대부분이므로 원천 의존성이 HTML/PDF로 고정

## 시나리오 5: 다국가 확장

- **상황:** 한국(DART) 외에 미국(SEC EDGAR), 일본(EDINET) 등 추가
- **변경 유형:** 노드 추가 + 규칙 변경
- **영향 범위:** 광역
- **난이도:** 높음
- **필요 작업:**
  1. 국가별 Entity 코드 체계 통합 (한국: 종목코드, 미국: CIK, 일본: EDINET 코드)
  2. 국가별 택소노미 매핑 (K-IFRS ↔ US-GAAP ↔ J-GAAP)
  3. 통화 차원 추가 (KRW, USD, JPY)
  4. 공시 주기·형식 차이 반영
  5. L3 정규화 계층에서 국가 간 비교 가능한 통합 Concept 정의
- **주의사항:** US-GAAP과 IFRS는 근본적 차이(예: LIFO 허용 여부)가 있어 단순 매핑 불가능

> **학습 반영 (onto_evolution):** XBRL 의미 식별자의 가치는 다국가 확장에서 극대화된다. 각국 택소노미가 XBRL 표준을 따르므로, XBRL 요소 ID를 보존하면 국가 간 매핑의 기초 앵커로 활용 가능하다.

## 시나리오 6: 정정 공시 처리

- **상황:** 이미 수집된 보고서가 정정 공시로 대체됨
- **변경 유형:** 원천 변경
- **영향 범위:** 국소
- **난이도:** 중간
- **필요 작업:**
  1. 정정 공시 감지 (동일 Entity-Period-Statement에 복수 인스턴스 존재)
  2. 최종본 판별 (접수번호·공시일시 기준)
  3. 이전 버전 FinancialFact를 비활성화(삭제가 아닌 버전 관리)
  4. 최종본 기준으로 참조 무결성(RI01, RI02) 재검증
  5. 시계열 질의(Q13) 시 정정 전후 차이 추적 가능하도록 이전 버전 보존
- **주의사항:** 정정 공시는 부분 정정일 수 있음 — 변경된 항목만 갱신하고 나머지는 원본 유지

> **학습 반영 (onto_dependency):** DART HTML이 유일 원천인 경우, 정정 공시 감지가 HTML 페이지 변경 감지에 의존한다. XBRL 인스턴스가 함께 정정되는지 여부가 불확실하므로, 원천 전환 규칙(dependency_rules.md)에 따라 검증 대상 원천을 결정한다.

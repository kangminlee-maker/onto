# Market Intelligence Domain — 확장 시나리오

## 1. 목적

시장 정보 분석 체계의 확장이 필요한 실무 시나리오를 정의하고,
각 시나리오에서 기존 구조가 어떻게 적응해야 하는지 명시한다.

## 2. 시나리오 목록

### EXT-01. 새로운 데이터 소스 유형 추가

**상황**: 기존 T1/T2/T3에 포함되지 않는 새로운 데이터 소스 유형이 등장한다.
예: AI 생성 시장 보고서, 소셜 미디어 실시간 센티먼트 데이터.

**확장 요건**:
1. source_tier 허용 집합에 새 값(예: T2-AI, T3-RT)을 추가한다.
2. `logic_rules.md` §2의 3지점 일치 규칙에 따라 수집/정책/프롬프트 3곳을 **동시에** 업데이트한다.
3. 새 tier에 대한 신뢰도 전파 규칙(`dependency_rules.md` §4.2)을 정의한다.
4. `competency_qs.md` Q1을 실행하여 정합성을 검증한다.

**구조적 영향**: source_tier 트레이트의 값 집합 변경. used_by에 명시된 모든 노드 타입에서 새 값이 유효하게 처리되는지 확인 필요.

### EXT-02. 정책 게이트 조건 추가

**상황**: 새로운 규제, 컴플라이언스 요구, 내부 정책에 의해 policy_gate에 새로운 통과 조건이 추가된다.

**확장 요건**:
1. policy_gate의 declared_conditions에 새 조건을 추가한다.
2. `logic_rules.md` §3의 선언-구현 연결 규칙에 따라 **즉시** 대응하는 검증 로직을 생성한다.
3. 검증 로직 없이 조건만 선언하는 것은 선언-구현 갭 위반이다.
4. 기존 StrategicOption 중 새 조건에 의해 gate_status가 변경되는 것을 식별한다.

**구조적 영향**: PolicyGate 노드의 조건 집합 확장. 기존 통과 전략의 재검증 필요.

### EXT-03. 위험 평가 프레임워크 교체

**상황**: ISO 31000 기반 프레임워크에서 COSO ERM 또는 자체 프레임워크로 전환한다.

**확장 요건**:
1. risk_score 산출 공식이 변경될 수 있다. `logic_rules.md` §5.1을 업데이트한다.
2. risk_level 트레이트의 허용 값이 변경될 수 있다. `structure_spec.md` §3을 업데이트한다.
3. 위험 평가의 **독립성**은 프레임워크 변경과 무관하게 유지되어야 한다.
4. 기존 RiskFactor 노드의 risk_score를 새 공식으로 재계산한다.

**구조적 영향**: 위험 평가 영역 내부 변경. 타 영역(분석, 전략 도출)에 대한 인터페이스(blocks, modifies)는 유지.

### EXT-04. 다중 도메인 동시 분석

> `onto_evolution` 학습: "다중 도메인 동시 분석"은 실무 요구이다.

**상황**: 하나의 프로젝트에서 market-intelligence와 다른 도메인(예: software-engineering, healthcare)을 동시에 적용한다.

**확장 요건**:
1. 모든 노드와 트레이트에 `market-intelligence::` 네임스페이스 접두어를 적용한다.
2. `concepts.md` §2의 동형이의어 테이블에 충돌 용어를 등재한다.
3. 교차 도메인 참조는 `{domain}::{concept}` 형식을 사용한다.
4. 도메인 간 의존 관계가 발생할 경우 `dependency_rules.md` §6의 규칙을 적용한다.
5. 각 도메인의 policy_gate는 독립적으로 운영된다. 교차 도메인 gate 통합은 별도 설계가 필요하다.

**구조적 영향**: 네임스페이스 체계 전면 적용. 기존 단일 도메인 참조를 네임스페이스 형식으로 마이그레이션.

### EXT-05. 실시간 모니터링 체계 추가

**상황**: 배치(batch) 분석에서 실시간(real-time) 모니터링으로 확장한다.
시장 변동, 경쟁사 동향, 위험 신호를 실시간으로 감지해야 한다.

**확장 요건**:
1. RawData에 `collection_mode` 트레이트를 추가한다: `{ batch, streaming, event-driven }`.
2. freshness_grade의 판정 기준을 실시간 데이터에 맞게 조정한다 (예: "expired"의 기준 시간 단축).
3. TrendSignal 노드에 `detection_latency` 속성을 추가한다.
4. 위험 평가의 risk_threshold 초과 시 **즉시** 알림을 트리거하는 경로를 설계한다.
5. source_tier 3지점 일치 규칙은 실시간 환경에서도 동일하게 적용된다.

**구조적 영향**: 트레이트 추가, freshness_grade 기준 변경. 기존 배치 분석 구조와의 호환성 유지 필요.

### EXT-06. 분류 축 세분화

> `onto_semantics` 학습: 분류 축 혼합 방지 원칙의 확장 적용.

**상황**: 기존 2축(노드 타입, 트레이트) 체계에 제3의 분류 축을 추가해야 한다.
예: "분석 관점(perspective)" — 같은 Market 노드를 기술적 관점, 재무적 관점, 규제 관점으로 다르게 분석.

**확장 요건**:
1. 새 분류 축을 `structure_spec.md`에 명시적으로 선언한다.
2. 새 축이 기존 노드 타입 축, 트레이트 축과 직교하는지 검증한다.
3. 축 간 혼합 금지 원칙을 새 축에도 적용한다.
4. `concepts.md` §3.1의 분류 축 분리 원칙을 업데이트한다.

**구조적 영향**: 분류 체계의 차원 증가. 모든 기존 노드에 새 축 값이 필요한지 여부를 결정해야 한다.

### EXT-07. 정량 분석 모델 통합

**상황**: 시장 규모 추정, 경쟁 시뮬레이션 등에 정량 모델(통계, ML)을 통합한다.

**확장 요건**:
1. 새 노드 타입 `AnalyticalModel`을 추가한다 (모델의 이름, 전제, 입출력 정의).
2. 관계 타입 `applies_model`: AnalyticalModel → Market/Competitor/Segment를 추가한다.
3. 모델 출력의 confidence는 입력 데이터의 source_tier와 모델 자체의 검증 상태에 의해 결정된다.
4. 모델이 policy_gate의 검증 로직으로 사용될 경우, 선언-구현 연결 규칙을 준수한다.

**구조적 영향**: 노드 타입 및 관계 타입 확장. 기존 신뢰도 전파 규칙에 모델 신뢰도 요소 추가.

## 3. 확장 시 공통 체크리스트

모든 확장 시나리오에서 아래 항목을 확인한다:

- [ ] source_tier 3지점 일치가 유지되는가? (`logic_rules.md` §2)
- [ ] 정책 게이트에 선언-구현 갭이 발생하지 않았는가? (`logic_rules.md` §3)
- [ ] 노드 타입과 트레이트의 분류 축이 혼합되지 않았는가? (`concepts.md` §3.1)
- [ ] 트레이트의 used_by와 값 집합 교차 검증을 통과하는가? (`structure_spec.md` §3.1)
- [ ] 위험 평가의 독립성이 유지되는가? (`domain_scope.md` §1)
- [ ] 다중 도메인 네임스페이스 충돌이 없는가? (`structure_spec.md` §5.4)
- [ ] 순환 의존이 발생하지 않았는가? (`dependency_rules.md` §5)
- [ ] `competency_qs.md`의 관련 질문을 재실행하여 통과하는가?

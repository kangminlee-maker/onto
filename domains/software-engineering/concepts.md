# Software Engineering Domain — 개념 사전 및 해석 규칙

## 아키텍처 핵심 용어

- Module = 모듈 → 독립적으로 배포/교체 가능한 코드 단위
- Interface = 인터페이스 → 모듈 간 상호작용의 계약. 구현과 분리되어야 함
- Dependency = 의존 → 한 모듈이 다른 모듈의 기능을 사용하는 관계
- Layer = 계층 → 관심사를 수직으로 분리하는 구조. 상위 계층만 하위 계층에 의존
- Bounded Context = 경계 맥락 → 특정 도메인 모델이 일관되게 적용되는 범위
- Source of Truth = 진실의 원천 → 불일치 발생 시 권위를 가지는 데이터/정의 소스. 이중화 시 우선순위 명시 필수

## 데이터/상태 관리 용어

- State = 상태 → 시스템의 현재 시점 정보
- Mutation = 변경 → 상태를 수정하는 연산
- Transaction = 트랜잭션 → 원자적으로 성공하거나 실패하는 연산 단위
- Migration = 마이그레이션 → 스키마 변경을 기존 데이터에 적용하는 절차
- Terminal State = 터미널 상태 → 더 이상 전이가 발생하지 않는 최종 상태. 이벤트 소싱에서 (1) 전이 이벤트, (2) Projector 분기, (3) 허용 후속 이벤트 3가지가 모두 정의되어야 함

## 타입 체계 용어

- Discriminated Union = 판별 합집합 → 공유 필드(discriminant)로 구분되는 타입 집합. string literal union과 별개의 타입 구성
- Exhaustive Check = 완전성 검사 → switch 문에서 모든 경우를 처리했는지 검증. `default: never` 패턴이 오류 위치를 원인 지점에 집중시킴
- Contract = 계약 → 모듈 간 약속. "무엇을 보장하는가"와 "무엇을 보장하지 않는가"를 모두 포함

## 제약 설계 용어

- Hard Constraint = 경성 제약 → 위반 시 시스템이 무효화되는 제약. 코드로 강제 가능
- Soft Constraint = 연성 제약 → 위반 시 품질이 저하되지만 시스템이 동작하는 제약. 문서/프로토콜로 권장
- Invariant = 불변식 → 시스템이 의미 해석 없이 구조만으로 탐지 가능한 제약. 코드로 강제
- Best-effort = 최선 노력 → 의미 해석이 필요하여 구조적 탐지가 불가능한 제약. 프로토콜로 권장

## 변경 관리 용어

- Breaking Change = 파괴적 변경 → 기존 소비자의 코드를 깨뜨리는 변경
- Backward Compatible = 하위 호환 → 기존 소비자가 수정 없이 사용 가능한 변경
- Deprecation = 폐기 예고 → 특정 기능을 유지하되 향후 제거를 예고

## 품질 용어

- Idempotent = 멱등 → 같은 연산을 여러 번 수행해도 결과가 동일
- Pure Function = 순수 함수 → 외부 상태에 의존하지 않고, 부작용이 없는 함수
- Deterministic = 결정론적 → 동일 입력에 항상 동일 출력

## 문서 설계 용어

- Contract Document = 계약 문서 → 코드와 1:1 대응하는 규격 문서. 위반이 곧 결함
- Guide Document = 가이드 문서 → 의도와 맥락을 기술하는 문서. 판단의 보조 자료
- Self-contained Spec = 자기 완결 명세 → 실행에 필요한 모든 정보를 자체 포함하는 명세. 특히 AI 에이전트가 실행 주체일 때 필수

## 주의가 필요한 동형이의어

- "service": 서비스 클래스(비즈니스 로직 담당) ≠ 서비스(배포 단위, 마이크로서비스)
- "model": 도메인 모델(비즈니스 객체) ≠ ML/LLM 모델 ≠ MVC 모델
- "controller": MVC 컨트롤러 ≠ 하드웨어 컨트롤러
- "context": 실행 맥락(런타임) ≠ 경계 맥락(DDD) ≠ React Context(상태 공유) ≠ 컨텍스트 윈도우(LLM)
- "migration": DB 마이그레이션 ≠ 시스템 마이그레이션(이전) ≠ source of truth 이전
- "validation": 입력 검증(사용자 데이터) ≠ 설계 검증(아키텍처 리뷰) ≠ AI 산출물 검증
- "constraint": 데이터 제약(DB) ≠ 설계 제약(아키텍처) ≠ 비즈니스 규칙

## 해석 원칙

- 코드의 현재 동작이 의도인지 버그인지는 코드만으로 판단할 수 없다. 테스트, 문서, 커밋 히스토리를 교차 확인한다
- "테스트가 통과한다"와 "올바르게 동작한다"는 같지 않다. 테스트가 검증하지 않는 영역이 존재할 수 있다
- 모듈 간 계약(타입 시그니처)은 "무엇을 보장하는가"와 "무엇을 보장하지 않는가"를 모두 명시해야 한다
- 설계 원칙이 합의되었을 때, 해당 원칙이 이미 다른 형태로 달성되어 있는지를 먼저 확인한다. 문자적 해석과 실질적 달성의 구분이 과설계를 방지한다
- "참조 무결성"(A가 B를 가리키는가)과 "의미적 정합성"(A 내용이 B 내용과 양립)은 별도 검증 계층이다. 전자가 통과해도 후자는 미보장

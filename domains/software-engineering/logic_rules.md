# Software Engineering Domain — 논리 규칙

## 타입 체계 논리

- 타입 시그니처가 선언한 계약은 런타임에 반드시 지켜져야 한다. 선언과 구현의 불일치는 논리적 모순이다
- optional → required 필드 변경은 breaking change이다. required → optional은 하위 호환이다
- 열거형(enum/union)에 값을 추가하는 것은 소비자 관점에서 breaking change이다 (exhaustive switch 깨짐)
- discriminated union과 string literal union이 다른 파일에서 동일 대상을 표현할 때, 한쪽 추가가 다른 쪽 컴파일 에러를 유발하지 않을 수 있다. `default: never` 패턴이 유일한 안전장치
- 실패 분기(success:false)에 상태 필드를 타입 수준에서 배제하면, 실패 시 부분 상태의 후속 전달을 컴파일 타임에 차단 가능

## 상태 관리 논리

- 상태 전이가 정의된 경우, 동일 입력에 대해 항상 동일 출력이어야 한다 (결정론)
- 부작용(side effect)이 있는 연산과 없는 연산을 혼합하면, 실행 순서에 따라 결과가 달라질 수 있다
- 비동기 연산에서 경합 조건(race condition)이 가능한지 확인해야 한다
- 이벤트 소싱에서 단일 비즈니스 작업이 복수 이벤트를 기록할 때, partial commit 방지를 위해 (1) 복합 이벤트 통합, (2) 선행 실행 후 마지막에 기록, (3) saga 패턴 중 하나를 명시적으로 선택해야 한다

## 제약 설계 논리

- hard/soft constraint 분류 도입 시, "무효화 여부" + "국소화 가능 여부" 2단계로 경계 사례를 처리한다
- 산출물 생성의 제약은 "체크리스트로 사후 검증"보다 "생성 지침에 사전 포함"이 더 강한 품질 보증을 제공한다. 사후 검증은 위반을 감지하지만 차단하지 못한다
- 자유 텍스트 pass_criteria와 expected_result는 AI 에이전트 실행 시 "부분 충족"과 "자의적 해석" 두 가지 장애 모드를 동시에 가진다

## 의존 논리

- 순환 의존이 존재하면 초기화 순서를 결정할 수 없다
- 인터페이스에 의존하는 것과 구현에 의존하는 것은 결합도가 다르다 (의존 역전 원칙)
- 하위 모듈이 상위 모듈에 의존하면 계층 역전이다

## 제약 상충 검사

- 두 규칙이 동일 대상을 서로 다른 방향으로 제약하면 상충이다
- 필수(required)와 선택(optional)은 독립 축이다
- 검증 규칙은 실행 순서에 무관해야 한다
- "참조 무결성"과 "의미적 정합성"은 별도 검증 계층이다. 참조 무결성 통과가 의미적 정합성을 보장하지 않는다

## 모듈 간 계약 논리

- 공개 API의 변경은 모든 소비자에게 영향을 준다
- 내부 구현의 변경은 공개 API가 유지되는 한 소비자에게 영향을 주지 않아야 한다
- 계약에 명시되지 않은 동작에 의존하는 것은 암묵적 결합이며, 위험하다
- 일괄 처리에서 Phase 단위로 트랜잭션 경계를 두고, 각 Phase 완료 시 정합성 검증을 수행하는 것이 실용적 패턴이다

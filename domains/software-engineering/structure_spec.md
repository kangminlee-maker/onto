# Software Engineering Domain — 구조 명세

## 모듈 구조 필수 요소

- **진입점(Entry Point)**: 사용자/외부 시스템이 접근하는 공개 인터페이스
- **비즈니스 로직**: 도메인 규칙을 구현하는 핵심 계층. 외부 의존 최소화
- **데이터 접근**: 저장소와의 상호작용을 추상화하는 계층
- **설정/환경**: 환경별 설정을 코드와 분리하여 관리

## 필수 관계

- 모든 공개 함수/클래스는 최소 1개의 호출자 또는 테스트가 있어야 한다 (고립 금지)
- 모든 외부 의존(라이브러리, API)은 인터페이스로 추상화되어 교체 가능해야 한다
- 설정 값은 코드에 하드코딩되지 않고, 환경 변수 또는 설정 파일에서 주입되어야 한다
- 구조적 검증(코드)과 실행 절차(프로토콜)가 다른 문서에 있을 때, 연결 고리를 프로토콜 문서에 역참조로 명시해야 enforcement가 완성된다

## 계층 구조 원칙

- 상위 계층(presentation/API)은 하위 계층(business/domain)에 의존한다
- 하위 계층은 상위 계층에 의존하지 않는다
- 동일 계층 내 모듈 간 의존은 최소화한다

## 권한과 계층 분리

- **정의 권한**(무엇이 있는가)과 **명세 권한**(어떻게 동작하는가)을 구분한다. 변경 방향은 정의→명세→코드
- 중간 추상 계층은 그 계층을 직접 소비하는 소비자가 2개 이상 존재할 때만 정당화된다. 소비자 1개이면 간접 비용만 발생
- "상세도의 종류"(의미/구조/구현)와 "상세도의 배치 방식"(계층 1/2/3)은 독립된 축이다. 혼동하면 불필요한 충돌이 발생
- "정의 문서"와 "명세 문서"의 경계 기준: "이것을 모르면 의도를 잘못 해석하는가?"

## 검증 구조

- 검증 계획을 AI에게 위임할 때, "무엇을 검증하라"(pass_criteria)와 "어떻게 검증했는지"(evidence_type)는 별도 필드로 분리해야 감사 추적이 가능
- edge case가 검증 계획에 열거되어도 결과를 기록하는 구조화된 경로가 없으면, "문서화된 edge case"와 "실제 검증된 edge case"가 분리된다
- "자동 실행 단계"와 "사용자 대화 단계"를 모두 포함하는 명령은, 상태 머신에서 다른 상태로 분리하면 세션 재개 복구가 단순화된다

## 고립 노드 금지

- 호출되지 않는 공개 함수/클래스 → 경고 (dead code)
- import되지 않는 모듈 → 경고 (고립 모듈)
- 테스트가 없는 공개 API → 경고 (검증 공백)
- 선언되었지만 도달 불가능한 상태(unreachable state) → 경고

## 저장/데이터 계층

- 데이터 소스(DB, API, 파일)와 비즈니스 로직은 분리되어야 한다
- 스키마 변경은 마이그레이션으로 관리되어야 한다
- 데이터 무결성 규칙(제약 조건)은 애플리케이션과 저장소 양쪽에서 검증되어야 한다

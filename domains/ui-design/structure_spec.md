# UI Design Domain — 구조 명세

## UI 설계 필수 요소

- **내비게이션 구조**: 정보 구조(IA), 글로벌/로컬/보조 내비게이션, 검색, 길찾기
- **입력 체계**: 폼 구조, 입력 유형별 패턴, 유효성 검증, 입력 보조
- **피드백 체계**: 시스템 피드백(성공/실패/경고/정보), 로딩 상태, 알림/토스트
- **상태 체계**: 빈 상태, 에러 상태, 로딩 상태, 온보딩 상태 — 정상 경로 외의 모든 상태
- **행동 유도 체계**: CTA 위계, 파괴적 행동 보호, 되돌리기 패턴
- **데이터 표시 체계**: 목록/테이블/카드/그리드, 정렬/필터, 페이지네이션
- **접근성 인터랙션**: 키보드 탐색, ARIA, 스크린 리더 대응, 시간 제한 관리

## 필수 관계

- 모든 사용자 행동은 대응하는 시스템 피드백이 정의되어야 한다 (행동-피드백 완전성)
- 모든 비동기 작업은 로딩/성공/실패 3가지 상태가 정의되어야 한다 (비동기 상태 완전성)
- 모든 입력 필드는 유효성 규칙과 에러 메시지가 정의되어야 한다 (입력-검증 완전성)
- 모든 데이터 표시에는 빈 상태가 정의되어야 한다 (빈 상태 완전성)
- 모든 에러 상태에는 복구 경로가 포함되어야 한다 (에러 복구 완전성)
- 모든 인터랙티브 요소는 키보드로 접근 가능해야 한다 (키보드 접근성)

## 화면 상태 매트릭스

모든 화면/컴포넌트에 적용되는 상태 목록:

| 상태 | 설명 | 필수 요소 |
|---|---|---|
| 이상적 상태 (Ideal) | 충분한 데이터, 정상 동작 | 핵심 콘텐츠, 주요 행동 |
| 최초 사용 (Zero/First-use) | 아직 데이터 없음 | 안내 메시지, 첫 행동 CTA |
| 부분 데이터 (Partial) | 일부 데이터만 존재 | 누락 영역 안내, 완성 유도 |
| 로딩 (Loading) | 데이터 요청 중 | 스켈레톤/스피너, 예상 소요 표시 |
| 에러 (Error) | 요청 실패 | 에러 원인, 복구 행동(재시도 등) |
| 빈 결과 (No Results) | 검색/필터 결과 없음 | 조건 변경 안내, 필터 초기화 |
| 권한 없음 (No Permission) | 접근 권한 부족 | 권한 요청 방법, 대안 경로 |
| 오프라인 (Offline) | 네트워크 연결 없음 (해당 시) | 캐시 데이터 표시, 재연결 안내 |

## 내비게이션 구조 패턴

| 패턴 | 적합 상황 | 부적합 상황 |
|---|---|---|
| 탑 내비게이션 (수평) | 주요 섹션 5~7개 이내, 데스크톱 중심 | 섹션 수가 많고 모바일 중심 |
| 사이드 내비게이션 (수직) | 섹션 수가 많거나 계층이 깊음, 관리 도구 | 모바일 환경, 콘텐츠 너비가 중요 |
| 바텀 내비게이션 | 모바일, 핵심 섹션 3~5개 | 섹션 수가 5개 초과, 데스크톱 |
| 탭 (수평/세그먼트) | 동일 맥락 내 2~5개 하위 뷰 | 탭 수가 많거나 탭 간 관계가 약함 |
| 브레드크럼 | 계층 3단계 이상, 상위 이동 필요 | 계층이 1~2단계로 얕음 |
| 햄버거 메뉴 | 부수 내비게이션, 공간 제약 | 핵심 내비게이션 (발견 가능성 낮음) |

## 피드백 유형-표현 매트릭스

| 피드백 유형 | 표현 패턴 | 지속성 | 위치 |
|---|---|---|---|
| 행동 성공 | 토스트 | 일시적 (3~5초) | 화면 상단 또는 하단 |
| 행동 실패 | 인라인 에러 + 배너 | 원인 해소까지 지속 | 해당 필드 근접 + 폼 상단 |
| 시스템 경고 | 배너 | 사용자 확인까지 | 화면 상단 |
| 정보 알림 | 토스트 또는 알림 센터 | 일시적 또는 읽음 처리까지 | 화면 상단 또는 알림 영역 |
| 로딩 (짧은) | 스피너 | 완료까지 | 행동 트리거 위치 |
| 로딩 (긴) | 스켈레톤 / 프로그레스 바 | 완료까지 | 콘텐츠 영역 |
| 파괴적 행동 결과 | undo 토스트 또는 확인 후 성공 메시지 | undo 가능 시간까지 | 화면 하단 |

## 폼 구조 원칙

- 필드 배치 순서: 가장 쉬운/빠른 필드부터 → 복잡한/민감한 필드 순. 초반에 완료감을 주어 이탈 방지
- 단일 컬럼 레이아웃이 원칙. 2컬럼 폼은 시선 이동 경로를 복잡하게 하여 필드 누락 위험 증가. 예외: 성/이름, 시/구 등 명확한 쌍(pair)
- 레이블 위치: 필드 위(top-aligned)가 가장 빠른 완료 시간. 좌측(left-aligned)은 스캔이 용이하나 완료 시간이 길어짐
- 입력 필드 너비는 예상 입력 길이를 반영해야 한다. 우편번호 5자리에 전체 너비 필드는 사용자에게 "더 많이 입력해야 하나?"라는 혼란을 줌
- 제출 버튼의 라벨은 구체적이어야 한다. "제출" 대신 "계정 만들기", "주문하기" 등 행동 결과를 명시

## 반응형 컴포넌트 전환 규칙

| 데스크톱 패턴 | 모바일 전환 | 전환 기준 |
|---|---|---|
| 데이터 테이블 | 카드 리스트 또는 핵심 열만 표시 | 컬럼 수 × 최소 열 너비 > 뷰포트 |
| 사이드 내비게이션 | 햄버거 메뉴 또는 바텀 내비게이션 | 뷰포트 768px 이하 |
| 수평 탭 (5개 이상) | 스크롤 가능 탭 또는 드롭다운 | 탭 전체 너비 > 뷰포트 |
| 멀티 컬럼 레이아웃 | 단일 컬럼 스택 | 뷰포트 768px 이하 |
| 모달 다이얼로그 | 전체 화면 또는 바텀시트 | 모달 너비 > 뷰포트 90% |
| 호버 툴팁 | 탭 팝오버 또는 인라인 텍스트 | 터치 환경 감지 |

## 도메인 경계 관리

- UI 설계 도메인은 "인터페이스가 사용자와 소통하는 구조"를 다룬다
- 시각적 표현(색상, 서체, 아이콘 스타일, 시각적 위계)은 visual-design 도메인 소관. 본 도메인은 "어디에 무엇을 배치하고 어떻게 반응하는가"에 집중
- UX 리서치(사용자 조사, 사용성 테스트, 퍼소나)는 별도 관심사. 리서치 결과는 UI 설계의 입력이 될 수 있으나, 방법론 자체는 본 도메인 소관이 아니다
- 프론트엔드 구현(React 컴포넌트, CSS, 상태 관리)은 software-engineering 도메인 소관. 본 도메인은 구현의 입력이 되는 인터랙션 명세를 제공
- 콘텐츠 작성(UX 라이팅, 마이크로카피)은 본 도메인과 중첩되나, 텍스트 내용 자체는 별도 관심사. 본 도메인은 텍스트의 위치, 시점, 구조를 다룬다

## 고립 노드 금지

- 비동기 작업 중 로딩/실패 상태가 정의되지 않은 것 → 경고 (상태 미정의)
- 데이터를 표시하는 화면 중 빈 상태가 정의되지 않은 것 → 경고 (빈 상태 부재)
- 에러 상태 중 복구 경로가 없는 것 → 경고 (막다른 길)
- 입력 필드 중 유효성 규칙이 없는 것 → 경고 (검증 부재)
- 인터랙티브 요소 중 키보드로 접근할 수 없는 것 → 경고 (접근성 위반)
- 내비게이션 경로 중 되돌아올 수 없는 것 → 경고 (복귀 경로 부재)
- 파괴적 행동 중 확인/되돌리기가 없는 것 → 경고 (보호 부재)
- UI 패턴 중 디자인 시스템/패턴 라이브러리에 없는 임의 패턴 → 경고 (시스템 외 패턴)
